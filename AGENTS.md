# Agents

## Project overview

Ansible project that provisions agent servers on ARM64 Ubuntu Server VMs managed by [lume](https://github.com/trycua/lume). Supports multiple agent stacks: [Hermes Agent](https://hermes-agent.nousresearch.com/), [OpenClaw](https://openclaw.org/), and [Claude Code](https://claude.ai/code). A single playbook (`configuration/deploy.yml`) deploys to any host using `--limit`.

Roles are provided by the [`tjmaynes.agents`](https://github.com/tjmaynes/ansible-agentic-collection) Ansible Collection. This repo is a consumer of that collection.

## Structure

```
configuration.example/       # Checked-in templates — copied by `make init`
  ansible.cfg
  deploy.yml
  hosts.yml
  vault.yml
  requirements.yml
configuration/               # Gitignored — user's actual config (generated by `make init`)
  ansible.cfg
  .vault_pass
  deploy.yml
  hosts.yml
  vault.yml
  requirements.yml
collections/
  ansible_collections/       # Installed collections (gitignored)
scripts/
  setup-vm.sh               # Idempotent lume VM creation script
  init.sh                   # Interactive setup — scaffolds configuration/ from templates
Brewfile                     # Homebrew dependencies (lume) — installed by `make install`
Makefile                     # Entry-point targets (install, init, setup, deploy, connect_hermes, ...)
```

## Key conventions

- Roles come from the `tjmaynes.agents` collection — installed via `make install` which runs `ansible-galaxy collection install -r configuration/requirements.yml`
- Playbooks reference roles with fully qualified names: `tjmaynes.agents.base`, `tjmaynes.agents.docker`, etc.
- The collection uses a shared `agent_name` variable to set the Linux service user, with `agent_*` prefixed variables for per-host config (e.g., `agent_discord_bot_token`, `agent_model`)
- Base role variables use the `base_*` prefix (e.g., `base_custom_dns_server`, `base_custom_ca_certificate_path`)
- Docker role variables use the `docker_*` prefix (e.g., `docker_honcho_enabled`, `docker_honcho_db_password`)
- Run `make init` to bootstrap `configuration/` — prompts for SSH user, vault password, host name, and IP
- `configuration/` is gitignored; `configuration.example/` is the checked-in template source
- All Makefile targets that touch Ansible set `ANSIBLE_CONFIG=configuration/ansible.cfg`
- One playbook (`configuration/deploy.yml`) targets all hosts; use `--limit <host>` to deploy individually
- Hosts are defined flat under `all.hosts` in `configuration/hosts.yml` — no group hierarchy needed
- Per-host variables (secrets, model config, optional features) are in inventory/vault; shared config is in play-level vars
- System-level tasks run as root (`become: true` at play level)
- Optional features are conditionally enabled via `is defined` checks — including custom CA certs, custom DNS, GitHub tokens, Discord, Home Assistant, and Mattermost integration
- All tasks must be idempotent — safe to re-run without side effects
- VM creation via `scripts/setup-vm.sh` is idempotent — skips if VM already exists
- `make install` runs `brew bundle` (to install `lume` from the `Brewfile`) before creating the Python venv and installing Ansible collections
- SSH into a deployed host as the hermes service user with `make connect_hermes HOST=<name>` (opens a tmux session)

## Secrets

Sensitive values use `no_log: true` and file mode `0600`. Always run `make encrypt` after editing `configuration/vault.yml`.

This repo is public — never commit plaintext secrets. Host IPs, tokens, and auth keys go in the encrypted vault inside `configuration/`.
